# QMT 背景与价值

这篇文档回答 4 个问题：

- QMT 是什么
- QMT 主要用来做什么
- 为什么它对 A 股量化特别方便
- 为什么 `quant-qmt` 要强调 Windows 隔离与安全边界

## 1. QMT 是什么

从迅投官方帮助中心的表述看，QMT 是迅投产品体系里专门面向 `极速策略交易` 的一条支持线。

但对 `quant-qmt` 这个项目要再说得更具体一点：

- 这里落地使用的运行链路，默认是 `MiniQMT`
- 官方 XtQuant 快速开始明确写到：`XtQuant 是基于迅投 MiniQMT 衍生出来的一套`，并要求先启动 `MiniQMT 客户端`
- `xttrader` 文档里的创建 API 示例，也直接要求传入 `MiniQMT 客户端 userdata_mini 的完整路径`

对开发者来说，更重要的是它周围那套配套接口：

- `xtdata`：行情与数据接口
- `xttrader`：交易与账户接口
- `xtconstant`：市场、价格类型、委托类型等常量
- 官方知识库与数据字典：说明如何下载数据、获取行情、查询持仓、处理回调

如果你做的是 A 股量化，这种“数据 + 交易 + 回调”接口在一个体系里，是非常实用的。

## 2. QMT 主要用来做什么

按官方文档，QMT / XtQuant 这套能力主要覆盖：

- 下载历史数据
- 获取实时行情
- 查询交易日、板块成分股、合约详情
- 发起报单和撤单
- 查询资产、委托、成交、持仓
- 接收订单、成交、资金、持仓等变动推送

从系统设计角度说，QMT 很适合担当：

- 券商连接层
- 本地执行层
- 行情接入层

而不是把整个研究平台、数据库、调度系统都耦合进来。

## 3. 为什么它对 A 股量化方便

### 3.1 Universe 和市场语义天然贴近 A 股

官方示例直接展示了：

- `get_trading_dates("SH")`
- `get_stock_list_in_sector("沪深A股")`
- `get_instrument_detail("000001.SZ")`

这意味着你做 A 股量化时，不需要先自己拼一层 universe 适配器，就能直接拿到：

- 交易日历
- 沪深 A 股股票池
- 合约基础信息

对于做全市场选股、行业轮动、日频策略、盘中监控，这一点非常省事。

### 3.2 常用数据获取成本低

对很多 A 股量化用户来说，QMT 的一个现实优点是：

- 常用数据链路已经在 QMT / xtdata 体系里
- 很多日常研究和执行需要的基础数据可以直接调
- 不一定一开始就要再接一套额外的第三方数据 SaaS

这里要说准确一点：

- 这不是说“所有数据都免费且无限制”
- 官方权限对比里明确区分了基础权限和 VIP 权限
- Level2、更长历史范围、更多品种或更高频能力，仍然要看权限和券商支持

所以更准确的说法是：

- 对已开通 QMT 的用户，很多常用 A 股数据可以直接用
- 更高阶数据能力再按权限逐步升级

### 3.3 数据周期对常见量化任务足够友好

官方数据文档里列出了这些基础周期：

- `tick`
- `1m`
- `5m`
- `1d`

同时也说明了很多更长周期是由基础周期合成得到，比如：

- `15m`、`30m`、`60m`
- `1w`、`1mon`、`1q`

这对 A 股量化很有用，因为多数策略其实就在几类数据上来回切换：

- 日频选股
- 分钟级择时
- 盘中风控
- 收盘前调仓

QMT 的数据接口并不神奇，但它离交易终端足够近，做“研究到执行”的闭环更顺。

### 3.4 可选五档全推和更细粒度行情

官方知识库写到，在券商 QMT 中可以配置 `五档全推`；数据字典中也列出了：

- `l2quote`
- `l2order`
- `l2transaction`

这里需要特别说明：

- 是否能用上这些更细粒度能力，取决于券商、版本和权限
- 不是所有个人账户都天然具备完整 Level2 能力

但从架构上讲，QMT 至少给了 A 股量化一个比较自然的升级路径：

- 先做日频 / 分钟级策略
- 再逐步接入盘口、逐笔、实时缓存

### 3.5 交易接口天然适合自动化执行

`xttrader` 官方说明里明确写到，它封装了 Python API，可与 MiniQMT 客户端交互，支持：

- 报单
- 撤单
- 查询资产
- 查询委托
- 查询成交
- 查询持仓
- 接收资金、委托、成交、持仓的主推消息

这意味着 A 股量化里非常常见的执行闭环可以自然搭起来：

1. 研究层产生信号
2. 执行层查询账户与持仓
3. 执行层下单/撤单
4. 回调层接收状态变化
5. 审计层落盘并复核

`quant-qmt` 就是把这条链路做成一个更容易独立部署的开源项目。

## 4. 为什么我们强调安全性

这里的“安全性”，最核心的一点其实很朴素：

- 策略代码尽量跑在你自己的环境里，而不是别人的服务器上

在这个前提上，再去谈：

- 策略运行边界清晰
- 券商终端隔离
- 凭证与接口暴露面可控
- 回调与执行过程可审计

不是指“交易一定安全”或“策略一定赚钱”。

### 4.1 官方特性本身就偏本地化

迅投知识库在聚宽迁移文档里写到：

- 所有数据都存储在本地
- 所有策略计算都在电脑本地运行

这是一个很重要的信号。对很多 A 股量化用户来说，本地化有几个现实好处：

- 更容易把券商终端和研究代码放在自己控制的环境里
- 不必把策略代码和交易逻辑直接托管到第三方云平台
- 更方便做本地落盘、日志、回调审计

这部分是官方事实；而“因此更适合做隔离部署”是我们基于这些事实做出的工程归纳。

### 4.2 策略迁移支持加密导入导出

官方帮助中心提到，QMT 支持策略或指标以 `加密模式` 导入导出。

这并不等于“万无一失”，但至少说明它在产品层面对策略迁移和本地策略保护是有考虑的。

### 4.3 Token 具备轮换失效语义

官方知识库提到：

- 接口 TOKEN 一次生成一个
- 刷新后前一个 TOKEN 失效

这对做数据接入、服务封装和权限轮换是有帮助的。它不是完整的零信任体系，但比“长期固定密钥到处复制”要健康得多。

## 5. 为什么 `quant-qmt` 要进一步收紧边界

基于上面的官方能力，我们在开源项目里进一步强调：

- Windows-only gateway
- 默认本机用 `127.0.0.1`
- 远程模式改 `0.0.0.0`，供 LAN / Tailscale / VPN / 内网调用
- 回调持久化
- 数据、交易、查询分层封装

这样做的目的不是把 QMT 变复杂，而是把它放到一个更适合协作、审计和远程调用的位置上：

- Windows 主机负责券商连接
- Linux / macOS / Docker 负责策略、任务编排和上层服务
- 网关成为唯一的交易入口

这对 A 股量化尤其重要，因为很多团队的真实约束就是：

- 券商终端必须在 Windows
- 策略开发不想被锁死在 Windows
- 交易入口必须尽量少、尽量可审计、尽量可控

## 6. 对开源项目的直接意义

所以 `quant-qmt` 不把 QMT 当成一个“顺手接上的小模块”，而是把它当成独立项目来建设：

- 独立仓库
- 独立安装
- 独立文档
- 独立 SDK / CLI
- 独立示例策略
- 独立部署边界

它最适合的角色不是“全能量化平台”，而是：

- A 股量化系统的 QMT 接入层
- Windows 上的本地执行边界
- 跨平台调用的统一网关
